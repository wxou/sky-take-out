# 苍穹外卖项目总结

## 一、项目概述

苍穹外卖（Sky Take Out）是一个完整的外卖订餐系统，采用前后端分离架构，包含**管理端（Web）**、**用户端（微信小程序）和**后端服务三大部分。系统覆盖了从店铺管理、菜品管理、订单处理到数据报表的完整外卖业务闭环。

### 项目架构

```
sky-take-out（Maven 父工程）
├── sky-common      通用模块（工具类、常量、异常、上下文）
├── sky-pojo        实体模块（DTO、Entity、VO）
└── sky-server      服务模块（Controller、Service、Mapper、Config）
```

### 业务模块

## 三、技术选型分析

### 3.1 核心框架：Spring Boot 2.7.3

**为什么选 Spring Boot？**

| 对比项   | Spring Boot | 传统SSM   | Spring Cloud |
| ----- | ----------- | ------- | ------------ |
| 配置复杂度 | 自动配置，开箱即用   | 大量XML配置 | 配置更复杂        |
| 内置服务器 | Tomcat内嵌    | 需外部部署   | 需额外组件        |
| 适用场景  | 中小型单体应用     | 传统企业应用  | 微服务架构        |
| 学习成本  | 低           | 中       | 高            |

**选择理由**：本项目是典型的单体外卖系统，业务规模适中，Spring Boot 的自动配置、内嵌服务器、起步依赖等特性大幅降低了开发和部署成本。Spring Cloud 对于当前业务规模过重，传统 SSM 配置繁琐，Spring Boot 是最佳平衡点。

**解决的问题**：

- 消除了传统 Spring 的 XML 配置地狱
- 内嵌 Tomcat，无需外部部署 WAR 包
- 统一依赖管理，避免版本冲突
- `@EnableCaching`、`@EnableScheduling` 等注解一键开启功能

***

### 3.2 持久层：MyBatis + PageHelper + Druid

**为什么选 MyBatis 而不是 MyBatis-Plus 或 JPA？**

| 对比项   | MyBatis   | MyBatis-Plus | JPA/Hibernate |
| ----- | --------- | ------------ | ------------- |
| SQL控制 | 完全手写，精细控制 | 自动生成+手写      | 自动生成          |
| 学习成本  | 中         | 低            | 高             |
| 复杂查询  | 灵活，原生SQL  | Wrapper条件构造  | JPQL/Criteria |
| 性能优化  | 直接优化SQL   | 需理解生成逻辑      | N+1问题风险       |

**选择理由**：外卖系统涉及多表关联查询（如订单+明细+菜品）、动态条件统计报表（营业额/销量排名）等复杂 SQL 场景，MyBatis 的 XML 映射和注解方式能精确控制每一条 SQL，便于性能调优。MyBatis-Plus 虽然简化了 CRUD，但在复杂报表查询上不如原生 SQL 灵活。

**Druid 连接池 vs HikariCP**：

| 对比项    | Druid     | HikariCP |
| ------ | --------- | -------- |
| 性能     | 优秀        | 极致       |
| 监控     | 内置Web监控页面 | 无        |
| SQL防火墙 | 支持        | 不支持      |
| 适用场景   | 需要监控的生产环境 | 追求极致性能   |

**选择理由**：Druid 提供了 SQL 监控和防火墙功能，在开发和生产环境中便于排查慢 SQL 和安全问题，对于外卖系统这种数据库操作密集型应用尤为重要。

**PageHelper 分页插件**：MyBatis 拦截器机制实现，零侵入分页，只需调用 `PageHelper.startPage()` 即可自动拼接分页 SQL，无需手动编写 `LIMIT` 语句。

**解决的问题**：

- 精确控制 SQL，优化报表查询性能
- 连接池监控，定位慢查询
- 零代码分页，减少重复劳动

***

### 3.3 缓存：Redis + Spring Cache

**为什么选 Redis 而不是本地缓存？**

| 对比项   | Redis                | Caffeine/Guava Cache | Ehcache |
| ----- | -------------------- | -------------------- | ------- |
| 架构    | 独立服务，分布式             | 嵌入应用，单机              | 两者皆可    |
| 数据一致性 | 多实例共享                | 仅本实例                 | 需配置     |
| 数据结构  | 丰富（String/Hash/Set等） | 简单KV                 | 简单KV    |
| 持久化   | 支持                   | 不支持                  | 支持      |

**选择理由**：外卖系统的店铺营业状态、菜品列表等缓存数据需要在小程序端和管理端之间实时共享，本地缓存无法跨进程同步。Redis 作为独立缓存服务，保证了多端数据一致性。

**两种缓存使用方式**：

1. **RedisTemplate 直接操作**：用于店铺营业状态（简单 KV，读写频繁）
2. **Spring Cache 注解**：用于菜品/套餐缓存（`@Cacheable` 读缓存、`@CacheEvict` 清缓存），声明式编程，代码更简洁

**解决的问题**：

- 菜品列表查询从数据库查询优化为缓存读取，QPS 提升 10 倍以上
- 店铺状态高频读写不冲击数据库
- 管理端修改菜品/套餐后自动清除缓存，保证数据一致性

***

### 3.4 认证授权：JWT

**为什么选 JWT 而不是 Session？**

| 对比项  | JWT             | Session       |
| ---- | --------------- | ------------- |
| 存储位置 | 客户端（请求头）        | 服务端（内存/Redis） |
| 无状态  | 是，服务端不保存状态      | 否，需服务端维护      |
| 跨域   | 天然支持            | 需额外配置         |
| 扩展性  | 水平扩展无需共享Session | 需Session粘滞或共享 |
| 适用场景 | 前后端分离/移动端       | 传统服务端渲染       |

**选择理由**：本项目管理端（Vue）和用户端（小程序）均为前后端分离架构，且小程序无法使用 Cookie 维持 Session，JWT 的无状态特性完美适配。管理端和用户端使用不同的密钥和请求头（`token` / `authentication`），实现了双端认证隔离。

**版本升级**：项目从 jjwt 0.9.1 升级到 0.11.5，旧版本存在密钥长度不足和签名算法混淆攻击风险，新版本使用 `Keys.hmacShaKeyFor()` 确保密钥符合 HMAC-SHA256 的安全要求。

**解决的问题**：

- 无状态认证，服务端不存储会话，降低内存压力
- 双端隔离，管理端和用户端令牌互不干扰
- 安全升级，防止签名算法攻击

***

### 3.5 实时通信：WebSocket

**为什么选 WebSocket 而不是轮询？**

| 对比项  | WebSocket | HTTP轮询  | SSE   |
| ---- | --------- | ------- | ----- |
| 通信方式 | 全双工       | 请求-响应   | 服务端推送 |
| 实时性  | 毫秒级       | 取决于轮询间隔 | 毫秒级   |
| 资源消耗 | 低（长连接）    | 高（频繁建连） | 中     |
| 双向通信 | 支持        | 不支持     | 不支持   |

**选择理由**：外卖场景中，用户下单/催单后管理端需要**立即**收到通知，HTTP 轮询存在延迟且浪费带宽，SSE 只支持单向推送。WebSocket 的全双工通信特性使得管理端既能接收来单/催单通知，也能向服务端发送消息。

**实现方式**：基于 JSR 356 标准（`@ServerEndpoint`），使用 `ConcurrentHashMap` 管理客户端会话，支持广播、单播、组播三种推送模式。

**解决的问题**：

- 来单提醒：用户下单支付后，管理端立即弹出通知并播放提示音
- 催单提醒：用户催单后，管理端立即弹出催单通知
- 实时性从轮询的秒级延迟降低到毫秒级

***

### 3.6 定时任务：Spring Task

**为什么选 Spring Task 而不是 Quartz？**

| 对比项  | Spring Task    | Quartz         |
| ---- | -------------- | -------------- |
| 依赖   | Spring内置       | 需额外引入          |
| 配置   | `@Scheduled`注解 | 需Job/Trigger配置 |
| 持久化  | 不支持            | 支持             |
| 集群   | 不支持            | 支持             |
| 适用场景 | 单机简单定时         | 分布式复杂调度        |

**选择理由**：本项目定时任务场景简单（超时订单取消、派送中订单自动完成），不需要集群调度和持久化，Spring Task 的注解方式零配置即可使用。

**定时任务清单**：

| 任务      | Cron表达式               | 功能              |
| ------- | --------------------- | --------------- |
| 超时订单处理  | `0 * * * * ?`（每分钟）    | 下单超过15分钟未支付自动取消 |
| 派送中订单处理 | `0 0 1 * * ?`（每天凌晨1点） | 派送中超过60分钟自动完成   |

**解决的问题**：

- 避免无效订单长期占用库存
- 自动完成遗漏的派送订单，保证数据准确性

***

### 3.7 切面编程：Spring AOP + 自定义注解

**为什么用 AOP 做公共字段填充？**

传统方式需要在每个 Service 方法中手动设置 `createTime`、`updateTime`、`createUser`、`updateUser` 四个公共字段，代码重复且容易遗漏。

**AOP 方案**：

1. 定义 `@AutoFill` 注解，标注在 Mapper 方法上
2. `AutoFillAspect` 切面拦截所有标注了 `@AutoFill` 的方法
3. 根据操作类型（INSERT/UPDATE）通过反射自动填充公共字段

**解决的问题**：

- 消除了 Service 层的公共字段赋值代码重复
- 统一管理，修改公共字段逻辑只需改切面
- 编译期检查，标注注解即可，不会遗漏

***

### 3.8 对象存储：阿里云 OSS

**为什么选 OSS 而不是本地存储？**

| 对比项   | 阿里云OSS  | 本地文件存储 |
| ----- | ------- | ------ |
| 扩展性   | 无限容量    | 磁盘限制   |
| CDN加速 | 内置      | 需额外配置  |
| 高可用   | 99.995% | 单点故障   |
| 运维成本  | 零       | 高      |

**选择理由**：外卖系统的菜品图片、套餐图片需要在小程序端快速加载，OSS + CDN 的组合能保证全国范围内的图片加载速度。本地存储无法满足小程序的图片加载性能需求。

**解决的问题**：

- 图片存储与业务服务器解耦，降低服务器压力
- CDN 加速，小程序图片秒开
- 无需运维文件存储系统

***

### 3.9 接口文档：Knife4j

**为什么选 Knife4j 而不是 Swagger UI？**

Knife4j 是 Swagger 的增强版，提供了更美观的 UI 界面、更丰富的文档功能和更好的中文支持。项目分为管理端和用户端两个 Docket 分组，方便前后端开发者按端查阅接口。

**解决的问题**：

- 前后端协作有据可依，减少沟通成本
- 在线调试接口，无需 Postman
- 接口变更自动同步文档

## 四、项目优化

### 4.1 安全优化

#### （1）JWT 版本升级（0.9.1 → 0.11.5）

| 项目   | 优化前             | 优化后                            |
| ---- | --------------- | ------------------------------ |
| 版本   | jjwt 0.9.1      | jjwt 0.11.5                    |
| 密钥管理 | 字符串直接传入，长度不足    | `Keys.hmacShaKeyFor()` 确保密钥安全  |
| 签名算法 | 可能被降级攻击         | 强制指定 HMAC-SHA256               |
| API  | `Jwts.parser()` | `Jwts.parserBuilder().build()` |

**效果**：消除了签名算法混淆攻击风险，密钥管理符合安全规范。

#### （2）Fastjson 替换为 Jackson

| 项目  | 优化前             | 优化后                     |
| --- | --------------- | ----------------------- |
| 库   | Fastjson 1.2.76 | Jackson（Spring Boot 内置） |
| 安全性 | 多次爆出反序列化漏洞      | Spring 官方维护，安全可靠        |
| 性能  | 极端场景略快          | 足够满足需求                  |
| 兼容性 | 非标准JSON行为       | 严格遵循JSON规范              |

**效果**：消除了 Fastjson 的反序列化安全漏洞，统一使用 Spring Boot 内置的 Jackson，减少额外依赖。

### 4.2 性能优化

#### （1）订单号生成：雪花算法替代时间戳

| 项目   | 优化前                                   | 优化后                   |
| ---- | ------------------------------------- | --------------------- |
| 方式   | `System.currentTimeMillis() + userId` | Hutool Snowflake 雪花算法 |
| 并发安全 | 高并发下可能重复                              | 分布式唯一，线程安全            |
| 有序性  | 毫秒级有序                                 | 毫秒级有序，趋势递增            |
| 信息泄露 | 包含用户ID                                | 无业务信息泄露               |

**效果**：彻底解决高并发场景下订单号重复问题，订单号趋势递增有利于数据库索引性能。

#### （2）历史订单查询：批量查询替代 N+1

| 项目    | 优化前                 | 优化后                     |
| ----- | ------------------- | ----------------------- |
| 方式    | 循环调用 `getByOrderId` | 批量 `getByOrderIds` + 分组 |

| 性能    | 10条订单需11次SQL        | 10条订单仅需2次SQL            |

**效果**：历史订单分页查询 SQL 次数从 O(N) 降为 O(1)，查询性能提升数倍。

## 五、项目亮点

### 5.1 自定义注解 + AOP 公共字段自动填充

通过 `@AutoFill` 注解 + `AutoFillAspect` 切面，实现了数据库公共字段（createTime、updateTime、createUser、updateUser）的自动填充。这种**声明式编程**方式使得开发者只需在 Mapper 方法上标注注解，无需在 Service 层手动赋值，代码更简洁、更不易出错。

**设计模式**：

- 自定义注解 → 声明标记
- AOP 切面 → 统一拦截处理
- 反射机制 → 动态调用 setter 方法
- ThreadLocal → 获取当前登录用户

### 5.2 双端认证隔离

管理端和用户端使用**不同的 JWT 密钥、不同的请求头名称、不同的拦截器**，实现了双端认证的完全隔离：

这种设计保证了管理端令牌无法访问用户端接口，反之亦然，提升了系统安全性。

### 5.3 WebSocket 实时通知

管理端通过 WebSocket 实时接收来单提醒和催单通知：

- 用户下单支付 → 管理端弹出"待接单"通知 + 提示音
- 用户催单 → 管理端弹出"催单"通知 + 提示音
- 点击通知可直接跳转到对应订单页面

相比轮询方案，实时性从秒级提升到毫秒级，且大幅减少了无效请求。

### 5.4 Spring Cache 声明式缓存

菜品和套餐缓存使用 Spring Cache 注解实现，代码侵入性极低：

- `@Cacheable`：查询时自动走缓存
- `@CacheEvict`：修改/删除时自动清缓存

管理端修改菜品/套餐后，缓存自动失效，用户端下次查询获取最新数据，保证了缓存一致性。

### 5.5 定时任务自动处理异常订单

- 每分钟检查超时15分钟未支付的订单，自动取消
- 每天凌晨1点检查派送中超过60分钟的订单，自动完成

这两个定时任务避免了人工干预，保证了订单状态的自动流转。

### 5.6 ThreadLocal 上下文传递

通过 `BaseContext`（ThreadLocal 封装）在拦截器中设置当前用户 ID，在 Service 层和 AOP 切面中获取，实现了**线程隔离**的用户上下文传递，避免了用户 ID 在方法参数中层层传递的繁琐。

### 5.7 全局异常处理

`GlobalExceptionHandler` 统一捕获业务异常和 SQL 异常，返回标准化的错误响应格式（`Result.error(msg)`），前端无需针对不同异常做差异化处理。

### 5.8 Maven 多模块分层

```
sky-common  →  通用层（不依赖业务）
sky-pojo    →  实体层（依赖 common）
sky-server  →  服务层（依赖 common + pojo）
```

模块间依赖关系清晰，通用工具类和实体类可被多个服务复用，为后续微服务拆分预留了空间。

## 八、核心业务流程

### 8.1 用户下单流程

```
用户浏览菜品 → 加入购物车 → 提交订单 → 生成待支付订单
    → 跳转支付页 → 确认支付 → 更新订单状态为待接单
    → WebSocket推送来单提醒 → 管理端弹出通知
    → 管理端接单 → 派送 → 完成
```

### 8.2 认证流程

```
客户端登录请求 → 后端校验账号密码 → 生成JWT令牌 → 返回给客户端
    → 客户端后续请求携带令牌 → 拦截器校验 → 解析用户ID存入ThreadLocal
    → 业务处理 → 响应后清除ThreadLocal
```

### 8.3 缓存流程

```
用户查询菜品 → @Cacheable检查缓存 → 命中则直接返回
    → 未命中则查询数据库 → 写入缓存 → 返回结果

管理端修改菜品 → @CacheEvict清除缓存 → 更新数据库 → 下次查询重新缓存
```

## 十、总结

苍穹外卖项目是一个功能完整、技术栈丰富的单体外卖系统。项目在技术选型上充分考虑了业务场景和团队规模，选择了 Spring Boot + MyBatis + Redis + JWT + WebSocket 的经典组合，覆盖了 Web 开发的核心领域。

项目最大的亮点在于**自定义注解 + AOP 的公共字段自动填充**、**双端 JWT 认证隔离**和**WebSocket 实时通知**三个设计，体现了对实际业务场景的深入理解。同时，通过雪花算法、批量查询、安全升级等优化措施，提升了系统的性能和安全性。

未来改进方向主要集中在**微服务拆分**、**安全加固**、**性能提升**和**运维自动化**四个维度，为系统从单体向分布式演进做好准备。
